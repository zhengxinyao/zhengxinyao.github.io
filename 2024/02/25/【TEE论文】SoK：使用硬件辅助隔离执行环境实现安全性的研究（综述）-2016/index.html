    SoK: A Study of Using Hardware-assisted Isolated Execution Environments for Security

    硬件供应商推出了各种 HIEE，包括系统管理模式、Intel 管理引擎、ARM TrustZone 和 Intel 软件防护扩展。这篇 SoK 论文对现有 HIEE 进行了全面研究探讨了 HIEE 的防御性和进攻性使用场景，并讨论了针对基于 HIEE 系统的攻击。

    1. INTRODUCTION

    隔离代码执行是实现安全性的基本方法之一。研究人员使用虚拟化技术来创建一个隔离的执行环境，用于运行防御性工具。此外，虚拟机自检[21]已被广泛用于攻击检测和恶意软件分析。然而，现有的基于虚拟化的方法存在局限性，包括：

    • 1）依赖于可能具有大型可信计算基础（TCB）的虚拟机管理程序。
    • 2） 无法处理虚拟机管理程序或固件 rootkit。基于虚拟化的方法依赖于虚拟机管理程序，因此它们无法分析相同或更高权限级别的 rootkit。
    • 3）遭受系统性能开销（例如，从虚拟机切换到虚拟机管理程序）

    隔离执行概念提供了一个可信执行环境 （TEE），使用硬件辅助技术将虚拟机管理程序排除在 TCB 之外，实现高级别的权限（即硬件级权限），并降低性能开销，从而在硬件中更快地执行上下文切换。

    在这篇 SoK 论文中研究了六种用于构建安全工具的硬件级计算环境（即 HIEE）。

    • 1） 传统 HIEE：系统管理模式 （SMM） 和动态测量信任根 （DRTM）
    • 2） 最近的 HIEE：Intel 管理引擎 （ME）、AMD 平台安全处理器 （PSP） 和 ARM TrustZone
    • 3） 最新的 HIEE：Intel Software Guard Extensions （SGX）。

    从两个方面确定了基于HIEE的系统的攻击和安全问题：1）隔离的计算环境（即HIEE本身）；2）使用HIEE进行安全的方法。
    请注意，HIEE 的概念需要与 TEE 区分开来。一方面，TEE 可能不是 HIEE，例如基于软件的 TEE（例如，虚拟化技术）;另一方面，HIEE 不一定是 TEE（例如，SMM 是 HIEE，但可能不是 TEE，因为它不是为安全而设计的）。

    这篇 SoK 论文的主要贡献是：

    • 我们对 SMM、Intel ME、AMD PSP、DRTM、Intel SGX 和 ARM TrustZone 等六种 HIEE 进行了深入研究，并比较了它们用于可信计算的硬件特性(2)
    • 我们探索了 HIEE 的防御性和进攻性使用场景，并使用最先进的基于 HIEE 的系统对其进行了描述(3)
    • 我们讨论了针对每个 HIEE 计算环境的所有攻击（例如，绕过隔离）和一些缓解措施(4)
    • 我们提出了对使用 HIEE 进行安全保护的方法的担忧，包括确保可信的交换路径和验证硬件技术的可信度(5)

    2. HIEE

    2.1 System Management Mode

    系统管理模式 （SMM） 是一种类似于 x86 平台上可用的 Real 和 Protected 模式的执行模式（英特尔从 90 年代初开始在其奔腾处理器中使用 SMM）。它提供了一个硬件辅助的隔离执行环境，用于实现特定于平台的系统控制功能，例如电源管理。它由基本输入/输出系统 （BIOS） 初始化。

    SMM 是通过在 CPU 上置位系统管理中断 （SMI） 引脚来触发的。
    此引脚可以通过多种方式断言，包括写入硬件端口或使用 PCI 设备生成消息信号中断。
    接下来，CPU 将其状态保存到称为系统管理 RAM （SMRAM） 的特殊内存区域。SMRAM 不能通过其他执行模式寻址。
    SMI 处理程序在引导时由 BIOS 加载到 SMRAM 中。SMI 处理程序可以不受限制地访问物理地址空间，并且可以运行特权指令（因此，SMM 通常称为Ring -2）。RSM 指令强制 CPU 退出 SMM 并恢复先前模式的执行。

    通常，有基于软件和硬件的方法来触发 SMI。
    在软件中，我们可以写入 ACPI 端口以引发 SMI。
    在基于硬件的方法方面，有许多硬件设备可用于引发 SMI，包括键盘、网卡和硬件定时器。图 1 显示了在保护模式下从操作系统触发 SMI 的过程。请注意，其他 CPU 模式（例如实模式）也可以通过触发 SMI 切换到 SMM。

    2.2 Intel Management Engine

    英特尔管理引擎 （ME） 嵌入在所有最新英特尔处理器中的微型计算机，包括服务器、工作站、台式机、平板电脑和智能手机。
    英特尔于 2007 年推出 ME 作为嵌入式处理器。当时，它的主要功能是支持英特尔主动管理技术（AMT），而英特尔AMT是第一个运行在ME中的应用程序。最近，英特尔开始使用 ME 作为可信执行环境 （TEE） 来执行安全敏感型应用程序。

    ME的硬件架构如图2所示。从图中我们可以看出，ME就像一台计算机；它包含处理器、加密引擎、直接存储器访问 （DMA） 引擎、主机嵌入式通信接口 （HECI） 引擎、只读存储器 （ROM）、内部静态随机存取存储器 （SRAM）、定时器和其他 I/O 设备。

    ME在处理器上执行指令，并具有代码和数据缓存，以减少对内部SRAM的访问次数。
    内部SRAM用于存储固件代码和运行时数据。
    ME还使用来自主系统存储器（即主机存储器）的一些动态随机存取存储器（DRAM）。
    此 DRAM 充当磁盘；ME处理器当前未使用的代码/数据的存储器页将从SRAM中逐出，并交换到主机存储器中的DRAM。DRAM 区域在系统启动时由 BIOS 保留。此 DRAM 专用于 ME 使用，操作系统无法访问它。 但是，ME 的设计不信任 BIOS，它假定主机可以访问保留的 DRAM 区域。

    由于嵌入式处理器（即ME处理器）无法对主机存储器进行寻址，因此引入了两个引擎（即DAM引擎和HECI引擎）来传输ME存储器和主系统存储器之间的数据传输。DMA 引擎用于在 ME 内存和主机内存之间移动大量数据。加密引擎用于执行昂贵的加密算法，包括 AES、SHA、DRING 和大数算术，ME 固件可以通过调用它们的 API 来使用它们。

    ME固件存储在两种类型的介质上：ROM和SPI闪存。如图 2 所示，ROM 位于 ME 中。ROM中的代码在制造阶段被烧录，无法修改。ROM 存储引导加载程序，并充当ME的信任根。

    大多数ME固件存储在SPI闪存中，它包括一个自定义操作系统和应用程序。根据ME中实现的应用程序，闪存固件在主机主板上的位置不同（例如，BIOS和NIC）。闪存通常由 OEM 锁定，以防止恶意修改。然而，研究人员已经证明绕过这些锁定机制将代码注入ME中。

    2.3 AMD Embedded Processors

    AMD 安全处理器是嵌入在 AMD 主 CPU 内部的专用处理器。它与 ARM TrustZone 技术和基于软件的可信执行环境 （TEE） 配合使用，支持运行第三方可信应用程序。

    AMD 安全处理器是一种基于硬件的技术，可实现从 BIOS 级别到 TEE 的安全启动。受信任的第三方应用程序能够利用行业标准 API 来利用 TEE 的安全执行环境。另一个例子是系统管理单元（SMU）。SMU 是 Northbridge 的一个子组件，负责在启动和运行时执行各种系统和电源管理任务。SMU包含一个处理器来辅助。由于AMD将北桥集成到CPU中，因此SMU处理器是CPU内部的嵌入式处理器，与Intel ME相同。

    2.4 Dynamic Root of Trust for Measurement

    Trust Computing Group （TCG） 在 2005 年的 TPM v1.2 规范中引入了动态信任根测量 （DRTM）。它是用于测量的静态信任根 （SRTM） 的替代方法。与在启动时运行的 SRTM 不同，DRTM 允许在任何时候初始化测量的信任根。

    为了实现这项技术，英特尔开发了可信执行技术 （TXT），提供了一种加载和执行系统软件（例如操作系统或 VMM）的可信方式。TXT 使用新的 CPU 指令 SENTER 来控制安全环境。英特尔 TXT 不会对系统状态做出任何假设，它为延迟启动提供了动态信任根。AMD有一种类似的技术，称为安全虚拟机SVM，TXT 和 SVM 都会在后期启动操作中引入大量开销。

    2.5 Intel Software Guard Extensions

    2013 年，英特尔发表了三篇关于软件防护扩展 （SGX） 的介绍性论文；SGX 是一组用于添加到英特尔架构处理器的内存访问的指令和机制。这些扩展允许应用程序实例化受保护的容器，称为 enclave。安全区可以用作 TEE，即使不信任 BIOS、固件、虚拟机管理程序和操作系统，它也能提供机密性和完整性。Jain等开发了OpenSGX，这是一个开源平台，通过修改QEMU在指令级别模拟英特尔SGX硬件组件。

    2.6 ARM TrustZone

    ARM TrustZone 技术是一种硬件功能，它提供了两种环境或世界。信任执行环境 （TEE） 称为安全世界，富执行环境 （REE） 称为正常世界。为了确保安全世界与正常世界之间的完全隔离，TrustZone 为硬件组件（包括 CPU、内存和外围设备）提供了安全扩展。启用了 TrustZone 的 ARM 平台上的 CPU 具有两种安全模式：安全世界和正常世界。

    图 3 显示了启用了 TrustZone 的 ARM 平台中的处理器模式。每种处理器模式都有自己的内存访问区域和权限。在正常世界中运行的代码无法访问安全世界中的内存，而在安全世界中执行的程序可以访问正常世界中的内存。安全世界和正常世界可以通过读取安全配置寄存器SCR中的 NS 位来识别。TrustZone 使用仅在安全环境中运行的监控模式作为管理两个世界之间交换机的看门人。正常世界可以调用称为安全监控调用 （smc） 的特殊指令进入监控模式并修改 NS 位以切换到安全世界。

    TrustZone 使用内存管理单元机制来支持安全和正常世界中的虚拟内存地址空间。两个世界中的同一虚拟地址空间映射到不同的物理区域。硬件中断有两种类型：中断请求 （IRQ） 和快速中断请求 （FIQ）。安全世界可以断言 FIQ 或 IRQ，而正常世界只能断言 IRQ。

    Summary of HIEEs

    3. USE CASES

    3.1 System Introspection

    系统自省已被广泛采用以确保安全性。有一系列工具使用 HIEE 进行系统自检、完整性检查和恶意软件检测。

    • HyperGuard 建议使用 SMM 来监控虚拟机管理程序的完整性，
    • HyperCheck也有类似的目标，但将快照外包给外部服务器进行操作系统/虚拟机管理程序完整性检查，因为它可以减少受保护机器上的计算开销。
    • HyperSentry使用带外通道，特别是智能平台管理接口，来触发 SMM 来检查对关键数据运行的基本代码的完整性。
    • Spectre 是另一个基于 SMM 的系统，它自检主机内存以检测恶意软件。它会定期检查主机内存是否存在堆溢出、堆喷射和 rootkit 攻击
    • SPROBES，它利用ARM TrustZone技术作为HIEE，将正常世界的内核执行限制在批准的内核代码内存中。它使安全世界能够使正常世界捕获任何正常世界的指令，并提供正常世界的处理器状态的不可伪造视图。
    • TZ-RKP是一个类似的基于 TrustZone 的系统，旨在在 ARM 平台上强制执行内核代码的完整性。它通过添加防御机制（包括数据完整性保护和性能增强）进一步改进了 SPROBES。

    3.2 Memory Forensics

    • jiang等提出了一种用于数字取证的固件辅助内存采集和分析工具。它利用 SMM 可靠地采集目标系统的易失性存储器，然后使用网卡将存储器内容传输到远程机器进行分析
    • SMMDumper在 QEMU 上实现了论文中提出的系统，它进一步支持转储和传输超过 4 GB 的物理内存
    • TrustDump 是一种基于 TrustZone 的内存获取机制，即使操作系统崩溃或遭到破坏，它也能可靠地获取移动操作系统的 RAM 内存和 CPU 寄存器。它使用 TrustZone 的安全域作为 HIEE，并在其中运行内存采集模块以进行内存取证。

    3.3 Transparent Malware Analysis

    Zhang等提出了MalT，一种用于恶意软件分析的裸机调试工具。其核心思想是使用 SMM 来增加调试的透明度。具体来说，它利用 SMM 作为 HIEE，在调试系统上留下最小的占用空间，并为调试器提供更透明的执行环境。我们相信，包括 ARM TrustZone 和 Intel ME 在内的其他 HIEE 也可用于透明的恶意软件分析，因为它们具有高权限和隐蔽性。此外，英特尔 ME 可以实现更高级别的透明度，因为它在协处理器上执行，并且不会在主 CPU 上引入任何性能开销。

    3.4 Executing Sensitive Workloads

    • Flicker和 TrustVisor使用 DRTM 和小型可信计算库来创建 HIEE。Flicker 使用 DRTM 创建按需安全环境，而 TrustVisor 使用 DRTM 安全地初始化轻量级虚拟机管理程序，该管理程序使用硬件虚拟化 （VT-x/SVM） 来保护在安全环境中运行的应用程序。这两个系统使用 TPM 提供远程证明并安全地存储数据以执行敏感工作负载。
    • Bumpy 是一个基于闪烁的系统，用于保护敏感的网络输入。它处理特殊代码模块中的输入，该模块使用闪烁在隔离环境中执行。
    • Sun等人提出了TrustICE，这是一个基于TrustZone的隔离框架，用于在移动设备上提供隔离计算环境（ICE）。

    3.5 Rootkits and Keyloggers

    研究人员已经使用硬件辅助的隔离执行环境来实现防御工具，但由于其高特权和隐蔽性，攻击者也可以将其用于恶意目的。

    SMM Rootkit：

    • 安全研究人员提议使用 SMM 来实施攻击。2004 年，Duflot 演示了第一个绕过 OpenBSD 保护机制的基于 SMM 的攻击。
    • Embleton等使用SMM实现了芯片组级键盘记录器和网络后门，该后门能够直接与网卡交互，通过网络数据包将记录的击键发送到远程机器。
    • 但是，这些攻击需要绕过或解锁 SMRAM 保护。

    ME Rootkits:

    • 服务器攻击已经使用 ME 进行了演示，以实现高级隐身 rootkit。
    • Tereshkin 和 Wojtczuk 将恶意代码注入英特尔主动管理技术 （AMT） 中，以实现 ME ring -3 rootkit。
    • DAGGER 是在 ME 中实现的基于 DMA 的键盘记录器，它在平台启动过程的早期捕获击键，这使得 DAGGER 能够捕获硬盘加密密码


    一方面，硬件辅助的隔离执行环境是强大的可信计算环境，用于执行安全功能（例如，如上所述的恶意软件/攻击防御的系统自检）;另一方面，它创造了一个理想的环境或基础设施，吸引攻击者实施超级强大的 Rootkit。

    5. DISCUSSION

    使用 HIEE 实现安全性的一个挑战是确保可信的交换路径。
    基于 HIEE 的系统假定攻击者具有环 0 权限;攻击者可以截获从正常环境到TEE的切换，并提供虚假的切换过程来欺骗用户（即欺骗攻击）。
    此外，攻击者可以通过简单地禁用由于其环 0 权限而禁用交换来对系统执行拒绝服务 （DoS） 攻击。
    幸运的是，Intel ME 和 AMD PSP 没有这个问题，因为它们独立于主 CPU 运行，因此没有切换过程。
    但是，其他 HIEE（如 SMM、DRTM、SGX 和 TrustZone）由于以时间切片方式共享主 CPU，因此存在此问题。

    使用 HIEE 实现安全性的方法在很大程度上取决于硬件供应商。它假设硬件供应商是值得信赖的，并且提供的硬件功能没有错误（例如，保证隔离）。不幸的是，没有明确的方法来验证这些假设。
    此外，由于各种原因（例如，商业保密），硬件供应商往往不会发布其实现的细节。
